# Cache Busting Rules

このサイトの手動運用向けキャッシュ回避ルール。

## 基本方針

- 同じファイル名のまま画像やHTMLを差し替えると、ブラウザに古い表示が残ることがある
- 差し替え時は必要に応じてキャッシュ回避を入れる
- むやみに全ファイルへ付けず、差し替え対象だけに使う

## 優先ルール

### 1. 画像を別用途で分けたいとき

- ファイル名自体を分ける
- 例
  - `_hash_xxx.webp`
  - `_hash_xxx_202601.webp`

この方法が最優先。

## 2. 同じファイル名を使い続けるとき

- URL に `?v=...` を付ける
- 例
  - `images/_hash_works_017.webp?v=20260416-3`
  - `images/_hash_portfolio_004.webp?v=20260416-2`

推奨書式:

- `?v=YYYYMMDD-N`
- 例: `?v=20260416-1`

## 3. works-data.json の画像差し替え

- 基本は `jacket` のファイル名を分ける
- 同一ファイル名のまま差し替える必要があるときだけ `?v=...` を付ける
- 同じ画像を複数作品で共有している場合は、片方だけ差し替えたいなら新ファイル名を作る

## 4. index / portfolio / works の直書き画像

- 差し替え直後に反映が怪しいときは `?v=...` を付ける
- 変更後は source と deploy の両方に反映する

## 5. デプロイ前チェック

- 差し替え画像の参照先が source と deploy で一致しているか確認
- 共有画像か個別画像かを確認
- 同じ曲名や同じ画像を複数箇所で使っていないか確認

## 6. ブラウザ確認

- 画像が変わらないときは、まずキャッシュを疑う
- 確認手順
  - ハードリロード
  - シークレットウィンドウ
  - 必要なら Chrome の DNS / socket キャッシュクリア

## このサイトでの実務ルール

- 作品画像差し替え:
  - できるだけ新しいファイル名を作る
- 一時的な表示不整合:
  - `?v=YYYYMMDD-N` を使う
- 修正完了報告前:
  - source
  - deploy
  - preview
  の3点を確認する
